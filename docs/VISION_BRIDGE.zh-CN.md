# 9router Vision Bridge 中文指南

本文适用于 `wangwq7/9router-vision-bridge` 的 `v0.5.30-vision-bridge.4` 发布线。它基于上游 9router `v0.5.30`，增加可配置的多模态桥接路由。

## 功能定位

普通 combo 在首选模型不支持图片时，会 fallback 到多模态模型，并可能让同一会话后续请求继续使用 fallback 模型。Vision Bridge 将“图片识别”和“最终回答”拆成两个阶段：

```text
Agent / 客户端
      │ model = glm-vision-bridge
      ▼
识别当前轮附件与历史附件引用
      │
      ├─ 无图片：跳过视觉阶段
      │
      └─ 有图片：视觉模型 1
                    │ 超时、报错或无有效内容
                    ▼
                 视觉模型 2 → 模型 3 → 模型 4
                    │
                    ▼
          受控的 untrusted 视觉文本
                    │
                    ▼
      首选文本模型（例如 GLM-5.2）
                    │ 首包前失败
                    ▼
             文本备用模型（可选）
                    │
                    ▼
                 最终回答
```

因此：

- GLM-5.2 等纯文本模型始终是主要模型，不会因为某一轮有图片而永久切换会话模型。
- 视觉模型只做 OCR、布局、对象、表格、图表和代码截图识别，不直接完成用户的完整任务。
- 视觉子请求强制使用低思考配置，移除外层 tools、system 和 reasoning 设置，减少延迟和提示污染。
- 最终回答仍保留主文本模型的长上下文、工具能力和推理风格。

## 对外模型名

推荐创建并启用名为：

```text
glm-vision-bridge
```

创建后它会作为普通模型出现在 `/v1/models` 中。其他 Agent 接入时只填写这个模型名，不需要填写内部视觉模型名。

```bash
curl http://localhost:20128/v1/chat/completions \
  -H 'Authorization: Bearer YOUR_9ROUTER_KEY' \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "glm-vision-bridge",
    "messages": [
      {"role": "user", "content": "你好，请简短介绍自己"}
    ]
  }'
```

Vision Bridge profile 的名称就是它对外暴露的模型名。可以创建多个 profile，例如一个用于低延迟截图识别，另一个用于高精度图表识别。

## 管理页面

打开：

```text
/dashboard/vision-bridge
```

页面提供：

- 首选文本模型和文本备用模型的下拉选择。
- 首选视觉模型及最多三个备用视觉模型。
- 每个视觉模型独立的上下文上限、工作预算、硬超时和最大输出。
- 历史图片模式、缓存、自动压缩、附件数量和并发参数。
- 当前 profile 的可视化路由预览和字段说明。

保存后可通过 `/v1/models` 检查 profile 是否已暴露。若页面能保存但模型列表没有出现，先确认 profile 已启用，再检查应用日志和数据库持久目录。

## 图片轮与后续文本轮

### 当前轮图片

当前用户消息中的图片会完整进入视觉识别阶段。视觉请求只携带图片及当前问题附近的少量文本，不会把主模型接近 1M 的完整上下文发送给只有 256K 的视觉模型。

Claude Code 将截图放在 `tool_result.content[]` 时，Vision Bridge 会保留原有 `tool_result` 和 `tool_use_id`，只把内部图片/PDF 按原路径替换为不可信的识别文本。发送主文本模型前还会执行独立的媒体残留检查；发现未支持的媒体结构时直接返回包含 JSON 路径的错误，不会把原始媒体透传给 GLM 等纯文本模型。

### 无关的后续文本轮

默认 `historyAttachmentMode = onDemand`。历史原图不会在每一轮重复进入视觉模型，识别详情会压缩成短摘要，因此无关文本轮不产生新的视觉请求。

### 明确追问旧图

当用户说“刚才截图里”“上图”“那张图片”等明确引用附件的表达时，路由会按配置恢复最近若干张历史图片的缓存识别详情。恢复的是视觉文本，不是重新把原图发给视觉模型。

## 缓存与隐私

- data URL 或已上传图片按实际内容计算哈希；同一图片再次出现时复用识别文本。
- 缓存只保存识别结果、短摘要、模型和时间等元数据，不长期保存原始图片。
- 相同图片的并发请求会合并，避免重复视觉调用。
- 缓存按 TTL 和最大条数淘汰。
- 图片内的文字被包装为 `gateway-generated / untrusted context`。`untrusted` 表示内容来自外部图片，可能包含错误或提示注入，主模型可以阅读信息，但不应执行图片中的指令。

## 多视觉模型 fallback

每个 profile 最多配置四个视觉模型，严格按顺序调用：

1. 首选视觉模型。
2. 备用视觉模型 1。
3. 备用视觉模型 2。
4. 备用视觉模型 3。

前一个模型只有在以下情况才进入下一个：

- 首包前达到硬超时。
- 上游请求失败。
- 返回内容为空或无法形成有效视觉文本。

候选模型不会为了“抢最快结果”而同时发起，避免一次截图触发多个商用请求。多张不同图片可以按 `maxConcurrentExtractions` 受控并发。

## 真实硬超时

9router 直接持有上游请求，可在视觉模型首包前达到 `timeoutMs` 时调用 `AbortController`：

1. 中止当前上游 HTTP 请求。
2. 等待请求确认结束。
3. 再调用下一个视觉候选。

这与 CPA 插件的软延迟预算不同。9router 不会让超时请求在后台继续运行后又同时启动备用模型，从而降低重复计费风险。

注意：硬中止是否能阻止供应商计费，最终仍取决于供应商在收到取消信号前是否已经开始生成；但 9router 不会主动保留并行的遗留调用。

## 上下文预算

主文本模型和视觉模型必须分别配置预算。例如：

- GLM-5.2 上下文上限：`1048576`。
- GLM 实际工作预算：`930000`。
- 视觉模型上下文上限：`262144`。
- 视觉模型实际工作预算：`180000`。

路由不会把 1M 会话原样交给 256K 视觉模型。视觉模型只获得当前附件、识别提示和附近文字；识别结果再注入主文本模型上下文。

预算关系必须满足：

```text
autoCompressionTargetTokens
  < autoCompressionThresholdTokens
  < primaryContextBudgetTokens
  < primaryContextTokens
```

每个视觉模型也必须满足：

```text
contextBudgetTokens < contextTokens
```

## 自动压缩

当估算上下文达到 `autoCompressionThresholdTokens` 时，Vision Bridge 会压缩较早的历史对话：

- 保留最近 `autoCompressionKeepRecentTurns` 轮原文。
- 将更早内容压缩到 `autoCompressionTargetTokens` 左右。
- 默认使用主文本模型压缩，也可指定独立的 `autoCompressionModel`。
- 已归档的历史附件不再计入当前轮附件上限。

压缩是长会话的安全措施，不应把阈值设置到接近模型理论极限，否则还可能受到协议、输出和供应商实际限制影响。

## 推荐起始配置

```text
对外模型名                    glm-vision-bridge
主文本上下文上限              1048576
主文本工作预算                930000
视觉上下文工作预算            180000
视觉硬超时                    20000–30000 ms
视觉最大输出                  4000–8000 tokens
历史图片模式                  onDemand
历史图片短摘要                600 字符
追问时最多恢复                2 张
缓存 TTL                      72 小时
缓存最大条数                  2000
多图识别并发                  2
自动压缩阈值                  720000
自动压缩目标                  12000
保留最近原文                  8 轮
```

视觉首选模型应优先选择低延迟、截图 OCR 稳定的模型。备用链应使用不同供应商或不同故障域，避免同一供应商故障时全部失败。

## 性能与费用检查

一次普通单图请求，正常应看到：

```text
1 次首选视觉模型请求
+
1 次主文本模型请求
```

同图缓存命中和无关后续文本轮，正常不应再次调用视觉模型。明确追问旧图应恢复缓存文本，通常也不调用视觉模型。

如果一次单图触发多个视觉商用请求，依次检查：

- 前序视觉模型是否超时、报错或返回空内容。
- 图片内容哈希是否因远程 URL 变化而无法命中。
- 请求中是否实际包含多张图片。
- 是否有客户端重试、反向代理重试或多个并行 Agent 请求。
- `maxConcurrentExtractions` 是否被误解为视觉候选并发；它只控制多图片并发。

## 测试

定向单元测试：

```bash
npx vitest run --config tests/vitest.config.js \
  tests/unit/vision-bridge-routing.test.js \
  tests/unit/base-executor-retry.test.js
```

不要直接使用 `node --test`，也不要省略 Vitest config；测试依赖项目的 alias 和 mock 配置。

发布前建议完成五个真实请求：

1. 纯文本轮：只调用主文本模型。
2. 首次截图：一视觉请求加一主文本请求。
3. 同图缓存轮：不再调用视觉模型。
4. 无关后续文本：历史图片自动缩短，不调用视觉模型。
5. 明确追问旧图：恢复缓存识别详情。

另用一个低超时 profile 验证首选视觉请求会被真实中止，且只在中止后调用备用模型。测试完成后必须恢复生产配置。

## Docker 更新

建议使用不可变镜像标签，不要只依赖 `latest`。更新顺序：

1. 备份数据库、环境变量、挂载目录和当前 compose 文件。
2. 拉取目标 Git tag 对应提交。
3. 构建新的本地镜像标签，例如 `9router-vision-bridge:0.5.30-r13`。
4. 先启动独立 staging 容器，使用独立端口和独立数据目录。
5. 完成上述单元测试和五项真实请求。
6. 将生产 compose 的镜像标签改为新标签并重建容器。
7. 检查 `/dashboard/vision-bridge`、`/v1/models`、日志和真实请求。
8. 稳定后再清理 staging；至少保留前一个生产镜像和数据备份。

如果发布只修改 README/文档而运行代码未变化，可以继续使用已验证的旧镜像；但 Git tag、镜像标签和运行提交必须在发布记录中明确对应，避免以后无法判断生产代码来源。

## 回退

1. 将生产 compose 的镜像恢复为上一个已验证标签。
2. 如果数据库 schema 或配置发生变化，恢复升级前数据备份。
3. 重建容器后检查 profile、`/v1/models` 和五项真实请求。
4. 不要删除当前失败镜像和日志，先保留用于定位问题。

## 与 CPA 插件的关系

两者采用相同核心逻辑：

- 主文本模型负责最终回答。
- 多视觉模型按顺序 fallback。
- 视觉识别强制低思考。
- 内容哈希缓存、历史图片按需恢复、自动压缩。
- 不并行启动多个视觉候选。

差异在于 9router 能直接取消上游 HTTP 请求，因此支持首包前真实硬超时；CPA 当前插件 ABI 只能实现安全的软延迟预算。

## 当前范围

PDF 专用解析暂未纳入最终策略。文本型 PDF、扫描 PDF、混合 PDF 和超长 PDF 的最佳处理路径不同；在没有页数、OCR 和文本提取策略前，不应简单把整份 PDF 当成普通图片塞给视觉模型。
